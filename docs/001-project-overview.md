## 1. 프로젝트 배경

### 1.1 MD(Markdown)의 근본적 한계

현재 AI 에이전트 지시 체계(GEMINI.md, system prompt, workflow 등)는 **Markdown 기반 자연어**로 작성된다. 이 방식에는 구조적 결함이 있다:

| 문제 | 설명 |
|------|------|
| **강제력 부재** | 프로그래밍 언어와 달리 컴파일러/인터프리터가 없어 규칙 위반 시 아무런 제재가 없음 |
| **비결정성** | 같은 MD 지시 → 매번 다른 결과물 생성 (확률론적 해석) |
| **검증 불가** | 스키마 검증, 타입 체크가 존재하지 않음 |
| **선택적 준수** | AI가 "읽고 싶으면 읽고, 말고 싶으면 말고" — 강제 이행 메커니즘 없음 |

```
코드:    규칙 위반 → ❌ 컴파일 에러 (실행 불가)
MD지시:  규칙 위반 → 🤷 아무 일도 안 일어남 (무시 가능)
```

### 1.2 QLN 1.0의 한계

현재 QLN(Query Language for N2)은 `qln.js`를 통한 **HTTP REST + CDP 브릿지** 구조로 동작한다:

- **단일 레이어**: 도구 호출 → HTTP POST `/call` → 결과 반환의 평면적 구조
- **SQL 부재**: 구조화된 쿼리 언어 없이 JSON 기반 도구 호출만 지원
- **시맨틱 검색 없음**: 도구를 이름으로만 찾음 (의미 기반 매칭 없음)
- **상태 관리 부재**: 워크플로우 간 상태 전이, 트랜잭션 개념 없음
- **보안이 블랙리스트 방식**: `SECURITY_PLUGINS` Set으로 차단 — 화이트리스트/역할 기반이 아님

### 1.3 실전에서 발견한 한계

워크플로우 제작 과정에서 확인된 구체적 문제점:
- AI가 `ai_surf` 대신 `ai_web_surf`를 호출하는 도구명 혼동
- 부팅 시퀀스를 MD에 적었으나 AI가 무시하는 현상 반복
- 워크플로우 단계 간 데이터 전달에 구조적 메커니즘 없음
- 에이전트 간 통신에서 메시지 포맷 불일치

## 2. 프로젝트 목표

### 2.1 핵심 비전

> **"AI 에이전트를 위한 진정한 프로그래밍 언어"** — 인간이 읽을 수 있으면서(MD), 기계가 강제할 수 있는(Rust), 데이터를 구조적으로 다루고(SQL), 의미를 이해하는(Llama Semantic)

### 2.2 4대 기술 축

```
┌─────────────────────────────────────────────┐
│               QLN 2.0 언어                    │
├──────────┬──────────┬──────────┬─────────────┤
│  Rust    │   MD     │  SQL     │ Llama       │
│  Core    │  Syntax  │  Query   │ Semantic    │
├──────────┼──────────┼──────────┼─────────────┤
│ 컴파일러  │ 가독성   │ 구조화   │ 의미 이해    │
│ 타입안전  │ 친숙함   │ 쿼리     │ 시맨틱검색   │
│ 강제실행  │ 확장성   │ 트랜잭션 │ 맥락 파악    │
│ 고성능   │ 템플릿   │ 상태관리 │ 유사도매칭   │
└──────────┴──────────┴──────────┴─────────────┘
```

| 기술 | 역할 | 핵심 가치 |
|------|------|----------|
| **Rust** | 컴파일러, 파서, 런타임 엔진 | 타입 안전, 강제 실행, 고성능 |
| **MD** | 문법 표면(Human-Readable Syntax) | 가독성, 친숙함, 기존 생태계 호환 |
| **SQL** | 구조화 쿼리 레이어 | 데이터 쿼리, 상태 관리, 트랜잭션 |
| **Llama Semantic** | 시맨틱 이해 레이어 | 의미 기반 검색, 맥락 파악, 유사도 매칭 |

### 2.3 QLN 2.0 핵심 목표

1. **강제력**: MD 규칙을 코드로 컴파일 → 위반 시 런타임 에러
2. **결정론**: 같은 `.n2` 파일 → 동일한 실행 결과 보장
3. **검증 가능**: 스키마 + 타입 시스템으로 사전 검증
4. **계층화**: 세분화된 레이어 시스템으로 관심사 분리
5. **시맨틱**: 도구/규칙/데이터를 의미 기반으로 검색 및 매칭

## 3. 기술 스택 개요

### 3.1 컴파일러 파이프라인 (Rust)

```
.n2 소스 → [Lexer] → [Parser] → [AST] → [Validator] → [Codegen] → 실행 가능 코드
                         ↓                    ↓
                    MD 호환 문법           타입 체크 + 스키마 검증
```

- **Lexer/Parser**: Rust의 `pest` 또는 `nom` 라이브러리 활용
- **AST**: Typst 방식의 마크업 + 프로그래밍 AST 하이브리드
- **Validator**: 스키마 기반 규칙 검증 엔진
- **Codegen**: MCP 도구 호출, 상태 전이 코드 생성

### 3.2 레이어 아키텍처 (QLN 2.0)

```
Layer 6: 🧠 Semantic Layer (Llama Embedding)
         의미 기반 도구 검색, 맥락 이해, 유사도 매칭

Layer 5: 📊 Query Layer (SQL)
         구조화 데이터 쿼리, 상태 관리, 트랜잭션

Layer 4: 🔒 Contract Layer (Rust Enforced)
         Agent Behavioral Contracts, 사전/사후 조건 강제

Layer 3: 📜 Rule Layer (Compiled MD → Rust)
         규칙 컴파일, 타입 검증, 스키마 체크

Layer 2: 🔌 Tool Layer (MCP + CDP)
         도구 바인딩, 보안 게이팅, 역할 기반 접근

Layer 1: 🌐 Transport Layer (HTTP/WebSocket/CDP)
         통신, 메시지 라우팅, 직렬화/역직렬화
```

### 3.3 시맨틱 엔진 (Llama)

- **Embedding**: `nomic-embed-text` 또는 Llama 계열 임베딩 모델
- **Vector Store**: SQLite + vector extension 또는 Qdrant
- **Hybrid Query**: SQL 정확 매칭 + 벡터 유사도 매칭 결합
- **용도**: 도구명 유사도 검색, 규칙 의도 파악, 워크플로우 추천

## 5. 선행 연구 참조

| 프로젝트/논문 | 관련성 |
|-------------|--------|
| **Typst** | Rust 기반 마크업 + 프로그래밍 언어 (가장 가까운 선행 사례) |
| **MDX** | MD + JSX 하이브리드 컴파일 (AST 변환 파이프라인 참고) |
| **mdschema** | MD 스키마 검증 (YAML 기반 선언적 검증) |
| **Agent Behavioral Contracts (ABC)** | AI 에이전트 계약 기반 거버넌스 (arxiv) |
| **AgentGuard** | 런타임 검증 프레임워크 (Dynamic Probabilistic Assurance) |
| **Open Agent Governance Spec** | 에이전트 거버넌스 표준 (결정론적 ID, 정책, 감사) |
| **MCP 2.0** | 구조화된 스키마 + 서버사이드 검증 |
| **LALRPOP/pest/nom** | Rust 파서 생성기/조합기 |
| **NL2SQL 하이브리드 엔진** | 자연어 → SQL 변환 + 벡터 검색 결합 |
